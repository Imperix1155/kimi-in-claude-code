## Post-release maintenance

- [x] **KMP-22** ✅ 2026-07-22 (v0.1.1) — task-mode output loss fix. A user bug report (Opus driving VRX) showed `/kimi:task` silently discarding completed work: when Kimi routes the deliverable through an "Agent" sub-agent / tool call and emits only a thin final message, that content was dropped from every output path, and an empty `end_turn` exited 0 as "completed". Root-caused empirically (live probes): sub-agent output arrives on OUR session (not child sessions — refuted the leading hypothesis) as ACP `{type:"content", content:{text}}` blocks that `extractTextContent` returned `""` for. Fixes: extractor unwraps the tool-content shape; `buildTurnResult` exposes `toolOutputs` + `hasContent`; `renderTaskResult` surfaces tool/sub-agent output (verbatim) and never prints stderr as the answer; exit status is content-aware (empty turn → nonzero → job `failed`); `--json` gains `toolOutputs` + `stderr`. Ran through the full review-loop (clean-context Claude + advisor + Codex/GPT-5.x): confirmed the core bug, corrected the report (prior assistant text is NOT wiped; the shell-verb allowlist is unsafe and was declined; rejection-feedback is already wired), and surfaced extra bugs now fixed (stderr-as-answer, whitespace-only exit-0) + a hardened test leak sweep (was matching the orchestration shell). Deferred/flagged: read-only shell allowlist = owner's security-policy call (declined by default); review-mode fan-out fails loudly (acceptable, untouched); non-text content blocks + messageId-reorder = latent LOWs. Verified: 7-suite battery + live reproduction (1594-char sub-agent deliverable surfaced through the real CLI, exit 0). _(from user bug report 2026-07-22)_
